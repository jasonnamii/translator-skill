# 다국어 번역 운영 도구

구조화된 문서(콜아웃, 테이블, 마크다운 꾸밈, 위키링크 포함 .md — apple-design-style 또는 html-div-style 활성시 div/HTML 포함)를 다국어로 번역할 때, **내용 번역보다 구조 일치가 더 자주 실패한다.** 이 스킬은 그 실패를 방지하는 운영 규칙과 QC 체크리스트를 제공한다.

---

## 번역 실행 워크플로우

```
원본(SoT) 확인 → 의도 주석 작성 → 번역 순서 결정 → 서브에이전트 배정
→ 1패스(충실 직역) → 2패스(네이티브 윤문) → 백트랜슬레이션 검증
→ 채점 루브릭 평가 → 구조 QC(qc_diff.py) → 편차 수정
```

### 번역 품질 향상 기법 (필수)

→ references/quality-techniques.md 참조

5대 기법(의도주석·네이티브표현사전·2패스번역·백트랜슬레이션·채점루브릭) 전체 풀적용. 워크플로우 통합 위치·실행 방법·판정 기준·Gotchas 모두 스포크 파일에 수록.

### 번역 순서 원칙

원본(SoT) → 영어(EN) → 나머지 언어 병렬. EN을 먼저 하는 이유: 영어가 가장 안정적이고, 구조 일치 검증의 기준점이 된다. 역방향 수정(번역본→원본)은 금지 — 반드시 원본을 먼저 갱신 후 파생물 동기.

### 공통 번역 원칙

| # | 원칙 | 이유 |
|---|------|------|
| 1 | **파일명은 영어** | 비해당어 사용자 핸들링. `_EN.md`, `_CN.md` 등 접미사 |
| 2 | **고유명사 보존** | 브랜드명·기관명·인명은 번역하지 않음. 현지어 설명은 괄호 병기만 |
| 3 | **주요 카피 영어 보존** | 프로그램명·섹션 타이틀·슬로건 등 브랜드 카피는 영어 원문 유지 |
| 4 | **금액 병기** | 원본 통화 + 괄호 현지 통화. 예: `400만원 (약 2.1만 RMB)` |
| 5 | **현지 맥락 괄호 추가** | 원본의 로컬 맥락을 번역 + 현지 대응 병기. 예: `韩国妈妈论坛（对应小红书）` |
| 6 | **톤 통일** | 격식체 기준 명시 후 전문 일관 유지. 번역자 재량에 의한 톤 변경 금지 |

## 번역 QC 체크리스트

번역 완료 후 **필수 실행**. 이 체크리스트를 건너뛰면 구조 편차가 그대로 전달된다.

| # | 검증 항목 | 방법 | 실패 사례 |
|---|----------|------|----------|
| 1 | **헤딩 수 일치** | `##`/`###` 카운트 = 원본 카운트 | 번역 중 서브섹션 누락 |
| 2 | **콜아웃 수 일치** | `[!type]` 카운트 = 원본 카운트 | ID에서 4개 콜아웃 누락 (평가 철학, 워크북, 학부모 등) |
| 3 | **div 스타일 일치** | `<div style="...">` 속성값 전수 비교 (apple-design-style 또는 html-div-style 활성시 적용. 마크다운 기본 문서에서는 N/A) | border·padding·radius 값 임의 변경 (2px→1px, 32px→16px) |
| 4 | **위키링크 유효성** | TOC `[[#heading]]` = 실제 헤딩 텍스트 | 헤딩 번역 후 TOC 동기화 누락 |
| 5 | **고유명사·카피 보존** | 원본의 고유명사·영어 카피가 번역본에서 변형 없이 존재 | 브랜드명 현지어 번역 |
| 6 | **파일 크기 편차** | 바이트 크기 ±25% 이상 → 내용 편차 의심 | TH +40% (UTF-8 멀티바이트 정상이나 확인 필수) |
| 7 | **내용 추가/삭제 없음** | 번역본에 원본에 없는 섹션·콜아웃 추가 금지 | ID에 원본에 없는 `[!abstract]` 신규 섹션 추가됨 |
| 8 | **격식체 일관성** | 언어별 지정된 격식체 기준과 일치 | 경어/반말 혼용 |

### QC 실행 방법

**1. 자동 검증 — `qc_diff.py` (필수)**

```bash
# 테이블 출력 (사람 보고용)
python3 ${CLAUDE_SKILL_DIR}/scripts/qc_diff.py <원본.md> <번역본.md>

# JSON 출력 (프로그래밍 연동용)
python3 ${CLAUDE_SKILL_DIR}/scripts/qc_diff.py <원본.md> <번역본.md> --json
```

자동 검증 커버리지 (13항목):

| 기본 검증 (카운트) | 확장 검증 (difflib/위치 특정) | 퍼지 검증 (의미적 감지) |
|-------------------|--------------------------|---------------------|
| 헤딩 수 (H1-H6) | 헤딩 시퀀스 정렬 — 누락·추가·레벨 불일치 위치 | — |
| 콜아웃 수 | — | — |
| 테이블 수 | — | — |
| div 개수 | div style 속성값 전수 비교 (border·padding·radius 등) | — |
| 코드블록 수 | — | — |
| 위키링크 수 | TOC `[[#heading]]` ↔ 실제 헤딩 교차 검증 | — |
| 이미지 임베드 수 | — | — |
| 리스트 아이템 수 (±10% 허용) | — | — |
| 파일 크기 비율 (±50% 경고) | — | — |
| — | — | **문단 대응 검증** — 타입별(text/heading/table/callout/html/code) 블록 수 비교. text는 ±20% 허용, 구조 요소는 정확 일치 |
| — | — | **미번역 잔류 감지** — 원본 문자 체계(한글/일본어/중국어/라틴/키릴)가 번역본에 8자↑ 연속 + 40%↑ 비율로 남아있는 줄 감지. 코드블록·위키링크·HTML 태그 내부는 제외 |

- exit code 0 = 전항목 PASS, 1 = FAIL 있음.
- `--json` 모드: `{"passed": bool, "summary": {...}, "extended_issues": [...]}` 반환.
- **FAIL 항목이 1건이라도 있으면 수정 후 재실행.** PASS 전까지 번역 완료 선언 금지.

**2. 수동 검증**: 격식체, 고유명사, 톤 → 서두 + 중간 + 말미 샘플링.

**3. 편차 발견 시**: 원본 기준으로 수정. 번역본 독자적 판단에 의한 구조 변경 불허.

## 서브에이전트 번역 운영 규칙

대규모 문서(800줄+, div/HTML 포함) 번역 시 서브에이전트 실패 빈도가 높다. 이유: 긴 HTML 구조를 유지하면서 텍스트만 교체하는 작업이 LLM에게 인지 부하가 크기 때문이다.

| # | 규칙 | 이유 |
|---|------|------|
| 1 | **1언어 1에이전트** | 병렬 처리 + 실패 격리. 한 에이전트가 2개 언어를 처리하면 두 번째 언어 품질이 떨어진다 |
| 2 | **구조 템플릿 선제공** | 문서 골격(콜아웃·테이블 구조, apple-design-style시 div/HTML 포함)을 에이전트에게 먼저 전달하고 '텍스트만 채워라'로 지시. 에이전트가 구조를 재창조하면 불일치 발생 |
| 3 | **섹션 단위 검증** | 전체 완성 후가 아니라 §1~§4, 부록 단위로 구조 일치 확인. 누락을 조기에 발견 |
| 4 | **실패 시 전체 재시도** | 부분 패치보다 전체 재작성이 품질 일관성 높음. TH 번역에서 부분 패치 시도 후 결국 전체 재작성이 더 빠르고 정확했음 |
| 5 | **정량적 완료 기준** | 헤딩 수·콜아웃 수·div 수가 원본과 100% 일치해야 완료 |
| 6 | **2패스 순차 실행** | 동일 에이전트가 1패스(직역)→2패스(윤문) 순차 수행. 1패스 건너뛰고 의역 직행 금지 |
| 7 | **역번역 별도 에이전트** | 번역 에이전트 ≠ 역번역 에이전트. 자기 번역을 자기가 역번역하면 원본 기억에 오염됨 |
| 8 | **채점 별도 에이전트** | 번역 에이전트 ≠ 채점 에이전트. 자기 평가 편향 방지. 4축 평균 ≥4.0 + 개별 ≥3.0 필수 |
| 9 | **언어당 호출 예산** | 풀패스: 번역(2패스) + 역번역 + 채점 = 에이전트 3~4회/언어. 5개 언어 = 15~20회 |

**수정4 적용:** 번역 산출물 편차 수정·부분 패치 시 trigger-dictionary §6 수정4 프로토콜 적용 — 레벨 판정→게이트→외과적 실행→POST_VERIFY(전파맵 포함). 특히 구조 편차 수정에서 old 잔존(잡탕)·중복삽입 방지를 위해 grep 잔존 검사 필수. 원본→번역본 순서 수정 시 전파맵으로 영향 범위 추적.

## 언어별 특성 레퍼런스

번역 대상 언어마다 고유한 함정이 있다. 새 언어를 추가할 때 이 테이블에 행을 추가.

| 언어 | 격식체 기준 | 길이 변동 | 주의사항 |
|------|-----------|----------|---------|
| **EN** | 서술형 격식체 | ±0% (기준) | 가장 안정적. 도메인 표준 용어 매칭만 주의 (IB/Oxford 등) |
| **CN (简体)** | 간체 공식 표기 | -20~30% | 짧아서 div 여백 불균형 가능. 간체/번체 혼용 절대 금지 |
| **JP** | 敬体(です・ます) | +10~20% | 경어·조사로 길어짐. 敬体 일탈(だ체 혼입) 주의 |
| **TH** | ภาษาทางการ | 바이트 +40% | UTF-8 멀티바이트 특성. 라인 수는 비슷하나 바이트 크기 큼. 서브에이전트 실패율 높음 |
| **ID** | EYD V 공식 표기 | ±5% | 콜아웃·div 구조 편차 빈도 가장 높음. QC 우선순위 1순위. 영어 차용어 범위 넓음 |
| **KO** | 합쇼체/해요체 | 원본 | 제출청소 시 격식체(합쇼체) 통일 확인 |

## Gotchas

- **가장 흔한 실패**: 번역 내용은 맞는데 구조(콜아웃 수, 테이블 구조, 위키링크)가 원본과 다름. apple-design-style 문서의 경우 div border/padding 값 불일치도 빈번. 에이전트가 "비슷하게" 만들기 때문. 반드시 속성값 전수 비교.
- **ID(인도네시아어) 특별 주의**: 5개 언어 중 구조 편차 빈도가 가장 높았음. 콜아웃 누락 + div 스타일 변경 + 원본에 없는 섹션 추가가 동시에 발생.
- **TH(태국어) 파일 크기**: 바이트 기준 +40%는 정상이지만, 매번 원인 확인 필요. "크기가 크니까 내용이 더 많겠지"로 넘기면 안 됨.
- **TOC 동기화**: 헤딩을 현지어로 바꾸면 TOC의 위키링크도 반드시 같이 바꿔야 함. 빠뜨리면 링크 깨짐.
- **역방향 수정 유혹**: 번역본에서 오류를 발견했을 때 번역본만 고치면 원본과 불일치. 반드시 원본→번역본 순서.
- **부분 패치 함정**: 서브에이전트가 80% 완성한 번역을 수동 패치로 마무리하면, 패치 부분의 톤·격식체가 달라짐. 차라리 전체 재시도가 낫다.
- **수정4 미적용**: 번역 편차 수정 시 수정4 프로토콜 없이 직접 패치하여 old 잔존(잡탕)·중복삽입이 발생하는 경우. 레벨 판정→게이트→외과적 실행→POST_VERIFY 필수.
- **qc_diff.py 미실행**: 눈으로 "비슷해 보인다"고 넘기면 div style 1px 차이, TOC 링크 깨짐을 놓침. 반드시 스크립트 실행 후 exit code 0 확인.
- **qc_diff.py FAIL 무시**: FAIL 항목이 있는데 "사소하니까 넘어가자"로 진행하면 안 됨. 특히 div style 불일치는 apple-design-style 문서에서 시각적 결함으로 직결.
- **의도 주석 생략**: "간단한 문서니까" 생략하면 톤이 뒤섞임. 최소 섹션 단위 의도 태그 필수.
- **2패스 역할 혼합**: 1패스에서 자연스럽게 쓰려 하면 정확성↓, 2패스에서 의미를 바꾸면 충실성↓. 역할 분리 엄격히.
- **채점 에이전트 = 번역 에이전트**: 자기 번역을 자기가 채점하면 관대해짐. 반드시 별도 에이전트.
- **네이티브 표현 사전 미갱신**: 프로젝트마다 새 도메인 표현이 나오는데 사전 미갱신 시 다음 번역에서 또 직역됨.
